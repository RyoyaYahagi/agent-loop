# Autonomous Dev Loop Ledger Lifecycle

This reference describes the expected lifecycle for a CI/LLM agent loop that uses `evidence-ledger.json` as the source of truth.

## Phase 1: Initialize

Create or update a ledger before implementation starts.

```bash
python scripts/ledger_init.py \
  --ledger evidence-ledger.json \
  --loop-run-id issue-123 \
  --repo owner/repo \
  --issue 123 \
  --branch feature/issue-123 \
  --base-ref main \
  --required-check test \
  --required-check lint
```

The ledger should contain loop scope, required checks, empty semantic lists, `machine_evidence`, `evaluations`, and `repairs`.

## Phase 2: Semantic planning

Record requirements, planned tasks, findings, and claims with `scripts/ledger_update.py`. These entries are annotations, not proof.

```bash
python scripts/ledger_update.py --ledger evidence-ledger.json --updates semantic-updates.json
```

Requirements should come from the full issue/user request, not only checklists.

## Phase 3: Machine evidence capture

Run checks through `ledger_run.py` so the ledger records command argv, cwd, branch, commit, exit code, timestamps, duration, and stdout/stderr logs.

```bash
python scripts/ledger_run.py --ledger evidence-ledger.json --check-id CHECK-test --type test -- pytest
```

Capture git and PR state when relevant.

```bash
python scripts/ledger_git_snapshot.py --ledger evidence-ledger.json
python scripts/ledger_pr_snapshot.py --ledger evidence-ledger.json --pr 123
```

## Phase 4: Deterministic evaluation

Run the evaluator. The evaluator is the pass/fail authority; LLM output is only annotation.

```bash
python scripts/evaluate_agent_loop.py --record --trigger local evidence-ledger.json
```

A run may only be reported as complete when the evaluator returns `PASS`.

## Phase 5: Repair

If evaluation fails, execute only the repair tasks generated by the evaluator. Mark repair state explicitly.

```bash
python scripts/ledger_repair.py --ledger evidence-ledger.json --repair-id REPAIR-001 --status started
# perform the repair
python scripts/ledger_repair.py --ledger evidence-ledger.json --repair-id REPAIR-001 --status completed --evidence CHECK-test
```

After a fix, re-run the relevant check and evaluator.

## Phase 6: Bounded CI controller

In CI, use `agent_loop_controller.py` for optional LLM repair. It must be bounded by attempts, repeated-failure count, and runtime.

```bash
python scripts/agent_loop_controller.py evidence-ledger.json \
  --max-attempts 3 \
  --max-same-failure-count 2 \
  --max-runtime-minutes 30 \
  --repair-command 'hermes chat -q "$(cat $HERMES_AGENT_LOOP_REPAIR_PROMPT_FILE)"' \
  --output-report agent-loop-handoff.md \
  --comment-pr
```

When limits are hit, the controller writes a human handoff report and exits with escalation instead of looping forever.

## Phase 7: Human handoff

The handoff report should include:

- stop reason
- attempts used
- repeated-failure count
- last verdict/score
- blocking failures
- last deterministic repair tasks
- relevant logs/evidence paths

In PR CI, post the report as a PR comment when credentials are available.
