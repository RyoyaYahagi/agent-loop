# PRレビュー用サマリー（人間向け）

> 目的: AI が作った PR を人間が短時間で判断できるように、見るべきポイントだけを日本語で集約する。
> 原則: AI の自己申告ではなく、CI・テスト・ledger・PR差分へのリンクを併記する。

## 1. このPRで解く課題

- Issue / 要望:
- 本当に解くべき課題:
- 今回のスコープ:
- スコープ外 / 後回し:

### 人間に見てほしいポイント

- [ ] 課題設定は正しいか
- [ ] スコープが広すぎない / 狭すぎないか
- [ ] 優先順位は妥当か

## 2. 設計判断

- 採用した方針:
- 代替案:
- 影響範囲:
- 将来変更への耐性:

### 人間に見てほしいポイント

- [ ] アーキテクチャ判断は妥当か
- [ ] 既存設計と矛盾しないか
- [ ] 将来の拡張・変更に弱くないか

## 3. 実装内容

- 主な変更ファイル:
- 責務分離:
- 既存設計との整合性:
- 注意すべき差分:

### 人間に見てほしいポイント

- [ ] 責務が適切に分離されているか
- [ ] 既存の設計・命名・データフローと合っているか
- [ ] 不要なリファクタや unrelated changes が混ざっていないか

## 4. テスト・CI

- 実行したローカルチェック:
  - [ ] test:
  - [ ] lint:
  - [ ] typecheck:
  - [ ] build:
- CI 状態:
  - [ ] CI green / required checks pass
  - [ ] CI failure あり（マージ禁止）
  - [ ] CI 未実行 / checks missing（マージ禁止）
- Evidence ledger:

### 人間に見てほしいポイント

- [ ] テスト観点は妥当か
- [ ] 重要シナリオ・境界値が漏れていないか
- [ ] CI が通る前にマージされていないか

## 5. リスク分類

| リスク | 有無 | 内容 | 人間判断が必要な理由 |
| --- | --- | --- | --- |
| 仕様逸脱 |  |  |  |
| 既存機能への影響 |  |  |  |
| データ/DB影響 |  |  |  |
| セキュリティ/RLS |  |  |  |
| パフォーマンス |  |  |  |
| 運用/ログ/監視 |  |  |  |

## 6. AIに任せたこと / 人間が判断すること

| 開発ステップ | AIに任せたこと | 人間が見るべきポイント |
| --- | --- | --- |
| 要件整理 | 論点洗い出し、質問作成、仕様ドラフト | 本当に解くべき課題、スコープ、優先順位 |
| 設計 | 代替案、影響範囲調査、ADR下書き | アーキテクチャ判断、将来の変更耐性 |
| 実装 | 小さな関数、テスト、リファクタ、ドキュメント | 責務分離、既存設計との整合性 |
| テスト | 単体/E2E生成、境界値追加 | テスト観点の妥当性、重要シナリオ漏れ |
| レビュー | 差分要約、リスク分類、レビュー観点提示 | 高リスク差分、仕様逸脱、非機能要件 |
| 運用 | ログ分析、障害仮説、修正案 | 本番影響、顧客影響、リリース判断 |

## 7. マージ前チェック

- [ ] PR は Draft ではない
- [ ] required CI がすべて成功している
- [ ] agent-loop evaluator が PASS している、または人間が FAIL 理由を明示的に許容した
- [ ] 高リスク差分の人間レビューが完了した
- [ ] 未解決コメント・残タスクが issue / follow-up に分離されている

## 8. 最終判断欄（人間用）

- [ ] マージ可
- [ ] 修正が必要
- [ ] スコープ再検討
- [ ] リリース判断が必要
